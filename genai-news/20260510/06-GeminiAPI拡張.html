RAGの土台が、画像とイベント駆動へ

File Search のマルチモーダル化と、Gemini API へのイベント駆動Webhooks追加

概要

Googleは2026年5月4日〜5日にかけて、Gemini APIの開発者向け機能を2系統で大きく強化した。File Searchツールには Gemini Embedding 2 による画像とテキストの統合埋め込み、カスタムメタデータによる絞り込み、ページ単位の引用という、本番RAGに必要な3点セットが追加された。 同時にバッチ・動画生成・非同期インタラクションの完了を 署名付きJSONでpush通知 するイベント駆動Webhooksも導入され、長時間ジョブのポーリングが不要となった。RAGとエージェント基盤を「Gemini API同梱」で一段引き上げる位置づけの更新だ。

File Search 3つのアップデート

マルチモーダル化・メタデータ・ページ引用と、本番RAGに必須の3点が一度に追加された。

マルチモーダル対応

Gemini Embedding 2 により画像とテキストを同一の埋め込み空間で扱える。スタイルや雰囲気で画像を検索する用途も。PNG / JPEG をサポート。

カスタムメタデータ

department: Legal のようなkey-valueを付与し、フィルタクエリでノイズを削減。RAGの精度と速度を改善。

ページ単位の引用

ページ番号を保持し、回答から具体的なソース位置に誘導可能。検証可能なRAG を志向。

対応フォーマット: PDF / Word / スプレッドシート / プレゼン / JSON / CSV / HTML / XML / Markdown / YAML / コード / ZIP / Jupyter Notebook、加えてマルチモーダルRAGでは PNG / JPEG。

システム構成イメージの比較

従来は複数のミドルウェアを自前で繋ぐ必要があったRAGが、Gemini API File Searchで 1つの呼び出し に集約される。

従来：自前RAGスタック

複数のSaaS／OSSを連結してパイプラインを組む。各段にコード・運用・コストが発生。

1. ファイル投入
PDF / Word / 画像
2. パーサ / チャンカー
Unstructured / LlamaParse
画像はOCR別経路
3. Embedding API
text-embedding系を別途呼出
画像用は別モデル
4. ベクトルDB
Pinecone / Weaviate / pgvector
メタデータ設計を自前で
5. 検索 / リランカー
ハイブリッド検索＋リランク
6. 引用情報の整形
ページ番号は自前で保持
7. LLM 呼び出し
プロンプトにcontext注入
-画像は 別経路 でOCR/キャプション化が必要
-メタデータフィルタの実装は ベクトルDB依存
-ページ番号引用は 自前でチャンク保持 する必要
-運用するコンポーネントが 5〜6個

新構成：Gemini API File Search 同梱

パーサ・Embedding・ベクトルDB・引用整形が File Searchツール内 に内包。アプリは2つのAPIを呼ぶだけ。

1. ファイル投入
PDF / Word / 画像 / コード等
2. File Search ツール（マネージド）
  • ✓パース＋チャンキング
  • ✓Gemini Embedding 2（テキスト＋画像を同一空間）
  • ✓ベクトルストア（メタデータ付き）
  • ✓カスタムメタデータフィルタ
  • ✓ページ単位の引用情報
3. Gemini モデル呼び出し
tools指定でFile Searchを利用
引用情報はレスポンスに同梱
+画像とテキストを 同一の埋め込み空間 で扱える
+key-valueメタデータ 付与とフィルタが標準機能
+ページ番号引用が レスポンスに自動付与
+運用するコンポーネントが 実質1個（ツール呼び出しのみ）

観点別の差分

観点 従来（自前スタック） 新構成（File Search）
画像対応 OCR / キャプション化を別経路で実装 Embedding 2 で テキストと統合
ベクトルDB Pinecone / pgvector等の運用が必要 マネージドで内包（運用不要）
メタデータフィルタ DB依存・スキーマ設計が必要 key-valueをそのまま付与
ページ単位引用 チャンクにページ番号を自前保持 レスポンスに 自動付与
コンポーネント数 5〜6個（パーサ／Embedding／DB／検索／LLM） 実質1個（File Search ツール）
実装コスト プロトタイプから本番に移すと急増 「週末プロトタイプ→本番」が連続
ロックイン 各コンポーネントを選択可能 Geminiエコシステム依存
トレードオフ: 自前スタックは 各レイヤを差し替えられる柔軟性 を残せる一方、運用負荷が高い。File Searchは 立ち上げの圧倒的な速さ と引き換えに、Geminiエコシステムへの依存が強くなる。アプリの寿命と差別化要件で選び分けるのが妥当。

このAPIの使いどころ

「マルチモーダルRAGを最初から作るならGemini File Search一択」ではない。RAGを簡単に作りたい場合に最も効くプロダクトとして位置づけるのが妥当。

向いているケース

  • +RAGを最速で立ち上げたい新規プロジェクト。ストア作成→ファイル投入で本番品質に届く
  • +メインLLMが すでにGemini で揃っている
  • +エンジニアリングリソースが限られ、ベクトルDB運用やパイプライン設計 を避けたい
  • +「画像の雰囲気・スタイル」での 類似検索 が主用途
  • +プロトタイプから中規模本番までを 一気通貫 で運ぶ

向かない（別の選択肢が良い）ケース

  • !既に Azure AI SearchやOpenAI Assistants API で動いている本番システム
  • !検索アルゴリズム（ハイブリッド検索・リランカー・スコアリング）を 自前でチューニング したい
  • !Embeddingモデルを 差し替えて比較 したい
  • !データ主権・コンプライアンス要件で 特定リージョン から出せない
  • !M365 / SharePoint等の 既存業務データに自動接続 したい（→ M365 Copilot / Copilot Studio）

類似プロダクトとの選び分け（参考）

プロダクト 立ち位置 特に強い領域
Gemini File Search マネージドRAG（API-first） 最速の立ち上げ、画像/テキスト統合の埋め込み
OpenAI Assistants API + File Search マネージドRAG（API-first） OpenAIで揃えたい場合の同等選択肢（テキスト中心）

一行まとめ: Gemini File Searchは 「マルチモーダルRAGを最も簡単に作るためのAPI」 として最適。本格的なチューニングや既存業務データ連携が要件になる規模では、Azure AI SearchやM365 Copilot系を併用・選択する。

イベント駆動Webhooks

長時間ジョブの結果取得が、ポーリングからpush通知方式へと刷新された。

Gemini APIに 署名付きJSON をプッシュ通知するWebhooksが追加された。バッチタスク・動画生成・非同期インタラクションなど、長時間ジョブのポーリング が不要になる。

従来（ポーリング）

client.sh
# 数秒ごとにステータス確認
while: GET /v1/operations/{id}
  sleep 5

新方式（Webhook）

server.ts
# 完了時にAPI側からpush
POST /webhook
  x-gemini-signature: ...
  { status: "done", ... }

開発者への意味

RAG基盤の標準化と、エージェントの非同期化という2つの実装トレンドが進む。

RAG実装の標準が一段引き上がる

画像の埋め込み、メタデータフィルタ、ページ単位引用は 本番RAGに必要な3点セット。これらが「Gemini API同梱」になることで、ベクトルDB周りを自前で組まなくても基本要件を満たせる。

エージェントの非同期化

動画生成や巨大バッチ推論を含む 長時間タスク をエージェントに任せる際、Webhookで完了を受け取れる。サーバーレス構成と相性が良く、ポーリング由来のコストとレイテンシを抑制。
