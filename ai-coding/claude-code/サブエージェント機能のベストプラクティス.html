レビューを挟むタイミングの正解

推奨：コミット直前 (Pre-commit)

「機能の実装が一通り動き、テストが通ったタイミング」でレビューエージェントを呼び出すのが最も対費用効果が高いです。

  • 文脈の切り替えコストを最小限に抑えられる
  • 人間がPRを出す前の「セルフチェック」の代わりになる
  • OWASPなどのセキュリティチェックを網羅的に行える

非推奨：プロンプトごとのレビュー

1行コードを書くたびにレビューを挟むのは、APIコストと時間の無駄になることが多いです。

  • コンテキストの読み込み直しが頻発する
  • スピード感が失われる
  • エージェントが「細かすぎる指摘」をしがちになる

作っておくべき「3人の賢者」

1. Strict Reviewer (書き込み権限なし)

Must Have

ファイル編集権限を剥奪し、「読むこと」と「批判すること」だけに集中させたエージェント。忖度のない指摘が得られます。

👍 推奨ツール設定: Bash (readonly) Grep Glob ※ Edit禁止
# .claude/agents/reviewer.md
name: code-reviewer
description: Expert code reviewer. MUST BE USED before any commit.
instructions: |
  あなたは厳格なシニアエンジニアです。
  コードを変更することは許可されていません。
  以下の観点でのみ指摘を行ってください：
  - セキュリティ脆弱性 (OWASP Top 10)
  - 可読性と命名規則
  - エラーハンドリングの欠落

2. Directory Architect (構造の番人)

Useful

コードを書く前に呼び出し、「どのファイルをどこに作るべきか」だけを相談するエージェント。プロジェクト構造の崩壊を防ぎます。

プロが教えるテクニック (Pro Tips)

魔法の言葉 "MUST BE USED"

エージェントの description フィールドに "MUST BE USED for all PRs" や "PROACTIVELY use this" と記述すると、メインエージェントが自律的にそのサブエージェントを呼び出す確率が劇的に上がります。

Gitコミット権限の委譲

git-committer というエージェントを作成し、コミットメッセージの生成と実行だけを任せると便利です。「Conventional Commitsを守って」と指示しておけば、履歴が非常に綺麗になります。

ドキュメント専任担当

コード修正が終わった後、最後に呼び出すのが doc-updater。コードの変更を読み取り、READMEやAPI仕様書だけを更新させます。人間が一番サボりがちなタスクを自動化できます。

出力形式の固定

サブエージェントへの指示で「結果は必ずJSON形式で出力して」と指定しておくと、メインエージェントがその結果をプログラム的に処理しやすくなり、連携ミスが減ります。
